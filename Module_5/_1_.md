# **1. Introduction to Requirements Engineering (RE)**

**Definition:**
Requirements Engineering is the **systematic process of eliciting, analyzing, documenting, validating, and managing the requirements of a software system**. It ensures the software **meets stakeholdersâ€™ expectations**.

**Key Points for Exam (~10 marks):**

1. **Purpose of RE:**

   * Understand what the user wants.
   * Reduce errors and cost by preventing misunderstandings.
   * Serve as a foundation for design, development, and testing.

2. **Stakeholders:**

   * People who have interest in the system.
   * Examples: end-users, clients, managers, regulatory bodies.

3. **Process vs Product Requirements:**

   * **Process:** Steps for developing software (methodology).
   * **Product:** Features and qualities of the software itself.

4. **Characteristics of Good Requirements:**

   * Correct, complete, consistent, verifiable, feasible, and understandable.

5. **Importance:**

   * Prevents software failures.
   * Ensures alignment with business objectives.
   * Facilitates communication among stakeholders.

6. **Example:**

   * Requirement: â€œThe online banking system shall allow users to transfer funds between accounts securely.â€
   * Shows **what** the system does (functional) and hints at **quality** (secure).

7. **Challenges in RE:**

   * Ambiguous requirements.
   * Conflicting stakeholder needs.
   * Changing requirements during development.

8. **Outcome of RE:**

   * Clear documentation: Use cases, functional and non-functional requirements, diagrams.

9. **Exam Tip:**

   * When asked â€œWhat is RE?â€, always include **definition, purpose, characteristics, and examples**.
   * Diagrams like Use Case can fetch extra marks.

10. **Remember:** RE = **Bridge between user needs & software solution.**

5. **Quality Attributes of Requirements:**

6. **Requirement Engineering Lifecycle:**

8. **Challenges:**

9. **Importance:**

10. **Example:**

# **3. Principles of Requirements Engineering**

**Key Principles (~10 marks):**

1. **Understand the problem first:** Donâ€™t start coding before knowing needs.
2. **Involve stakeholders continuously:** Users should review requirements.
3. **Document requirements clearly:** Avoid vague words.
4. **Prioritize requirements:** Not all requirements are equally important.
5. **Manage changes carefully:** Track revisions and updates.
6. **Consistency:** Avoid conflicting requirements.
7. **Traceability:** Every requirement should be linked to a source or objective.
8. **Feasibility:** Requirements must be realistic technically and financially.
9. **Validation:** Always verify requirements with stakeholders.
10. **Example:** In e-commerce, â€œThe system should support up to 10,000 users simultaneouslyâ€ is a **high-priority non-functional requirement**.

# **5. Types of Requirements**

Requirements are **different kinds of statements about the system**. Understanding them clearly helps in exams.

**1. Functional Requirements (FRs):**

* Define **what the system should do**.
* Example: â€œThe system must allow users to log in with username and password.â€
* Exam tip: Focus on actions, features, or services.

**2. Non-Functional Requirements (NFRs):**

* Define **how the system should behave** (quality or constraints).
* Examples: performance, security, usability, reliability.

  * â€œPages should load in 2 seconds.â€
  * â€œPasswords must be encrypted.â€

**3. Domain Requirements:**

* Specific to the **industry or business rules**.
* Example: Banking system must comply with SWIFT transactions.

**4. User Requirements:**

* Written in **simple language** for end-users.
* Example: â€œI want to search products by category.â€

**5. System Requirements:**

* Detailed and **technical**, used by developers.
* Example: â€œSystem shall use MySQL database for storing user data.â€

**6. Interface Requirements:**

* Define how system **connects with other systems**.
* Example: â€œThe system should send data to the payment gateway via REST API.â€

**7. Performance Requirements:**

* How fast or responsive the system is.
* Example: â€œThe website should handle 10,000 concurrent users.â€

**8. Safety & Security Requirements:**

* Protect users and data.
* Example: â€œData must be encrypted using AES-256.â€

**9. Maintainability & Scalability Requirements:**

* How easy it is to maintain or upgrade.
* Example: â€œNew modules should be addable without downtime.â€

**10. Exam Tip:**

* Always **classify requirements** as functional/non-functional/domain/user/system.
* Use **examples** in answers; it gives extra marks.

**The RE process has 6 main activities:**

5. **Management:** Track changes, priorities, and versions of requirements.

6. **Negotiation:** Resolve conflicts between stakeholders.

# **1. Introduction to HTTP and HTTPS**

**HTTP (Hypertext Transfer Protocol):**

* Protocol used by **web browsers and servers** to communicate.
* Works on **request-response model**:

  * Browser sends request â†’ Server responds with data (HTML, images, etc.)
* **Port:** 80
* **Not secure**: Data is sent **in plain text**, vulnerable to interception.

**HTTPS (Hypertext Transfer Protocol Secure):**

* **Secure version of HTTP**.
* Uses **SSL/TLS encryption** to protect data.
* **Port:** 443
* Prevents **eavesdropping, data tampering, and man-in-the-middle attacks**.

**Key Features:**

| Feature  | HTTP            | HTTPS                             |
| -------- | --------------- | --------------------------------- |
| Security | No encryption   | Data encrypted with SSL/TLS       |
| Port     | 80              | 443                               |
| Speed    | Slightly faster | Slightly slower due to encryption |
| Use      | Basic websites  | Banking, e-commerce, login forms  |

**Example:**

* HTTP: `http://example.com` â†’ Not secure
* HTTPS: `https://example.com` â†’ Secure, padlock icon in browser

**Exam Tip:**

* Always **mention SSL/TLS and encryption** for HTTPS.
* Explain **why HTTPS is important** for sensitive data.

---

# **2. HTTP vs HTTPS**

**Differences (simple table for exam clarity):**

| Feature         | HTTP                | HTTPS                             |
| --------------- | ------------------- | --------------------------------- |
| Security        | None                | Encrypted via SSL/TLS             |
| URL Prefix      | http://             | https://                          |
| Data Protection | No                  | Yes (confidentiality & integrity) |
| Performance     | Faster              | Slightly slower                   |
| Common Use      | Informational sites | Banking, e-commerce, login forms  |
| Port            | 80                  | 443                               |

**Key Point:** HTTPS = HTTP + Security
